# Repository Guidelines

## Project Structure & Module Organization

See [`ARCHITECTURE.md`](ARCHITECTURE.md) for the repository layout, crate responsibilities, WIT organization, and
phase-level design context. Treat it as the canonical source before changing module boundaries or introducing new
top-level directories.

## Build, Test, and Development Commands

- `cargo install-tools`: installs the CI toolchain from `.cargo/config.toml` (`cargo-nextest`, `cargo-llvm-cov`,
  `cargo-deny`, `wasm-tools`, and related tools).
- `cargo build --workspace`: builds all workspace crates.
- `cargo nextest run --workspace --all-features`: runs the workspace test suite; prefer this over `cargo test`.
- `cargo fmt --all`: formats Rust code.
- `cargo clippy --all-targets -- -D warnings`: runs lints with warnings treated as errors.
- `cargo deny check` audits licenses, advisories, bans, and sources using `deny.toml`.
- `cargo machete` checks for unused dependencies before dependency cleanup.
- `wasm-tools component wit wit/oxidhome.wit`: validates the WIT surface.

## Coding Style & Naming Conventions

Use Rust 1.95.0, edition 2024, as pinned in `rust-toolchain.toml`. Follow `rustfmt` output. The workspace denies
`unsafe_code`; any explicit opt-in must include a focused `// SAFETY:` justification. Prefer `tracing` for logging, not
`log`. Use `anyhow` in binaries and host-internal crates, and typed errors for externally consumed library APIs when
introduced. Public items should have useful doc comments.

Prefer descriptive snake_case for modules, functions, and tests. Prefer direct sibling modules such as `host_impl.rs`
over `mod.rs`. Public structs and enums use `UpperCamelCase`.

Keep exact-pinned dependency bumps isolated for crates that affect generated bindings or runtime embedding, especially
`wit-bindgen` in `oxidhome-wit` and `wasmtime` in `oxidhome-core`.

## Testing Guidelines

- Put crate-level integration tests in `crates/<crate>/tests/`; existing host tests are in
  `crates/oxidhome-core/tests`.
- Name tests after the behavior under test, e.g. `simulated_switch.rs` or `event_dispatch.rs`.
- For plugin/host flows, use the existing support helpers that build example plugins for `wasm32-wasip2`.
- Use `#[tokio::test(flavor = "current_thread")]` for tests that rely on thread-local `tracing` subscribers.
- Integration tests that shell out to `cargo` should use the `spawn_clean_cargo` helper in
  `crates/oxidhome-core/tests/support.rs`. It strips `RUSTC_WRAPPER`, `RUSTC_WORKSPACE_WRAPPER`, and
  `cargo-llvm-cov` instrumentation env vars so nested `cargo build --target wasm32-wasip2` calls do not inherit
  `-Cinstrument-coverage`.
- Target 85% line coverage per crate. CI tracks coverage with `cargo llvm-cov nextest`; drops below target should be
  justified in the PR description.

## Commit & Pull Request Guidelines

**Never commit to `main` directly.** Always create a branch, push it, and open a pull request — even for small
docs / chore changes. `main` is updated only by merging PRs.

**Branch names** mirror the commit-prefix vocabulary: `feat/<slug>`, `refactor/<slug>`, `fix/<slug>`, `docs/<slug>`,
`test/<slug>`, `chore/<slug>`. Pick the prefix that matches the dominant change in the branch.

**Commits** use concise, scoped prefixes such as `docs:`, `loader:`, `test:`, `spec:`, `fixups:`. Follow that style
with an imperative summary, for example `loader: preserve base URL for nested refs`. **Always pass `-s`** so each
commit carries a `Signed-off-by` trailer. Do not use `Co-authored-by` for agent assistance; use `Assisted-by`
instead. Do not include `Generated by Agent` in commit messages or pull request descriptions.

**Pull request descriptions** should be the main "what changed / why" content only. Cover the behaviour change,
call out breaking changes for downstream consumers when applicable, and link related issues. Include fixture names
or minimal JSON examples for parser / validator changes so reviewers can reproduce the case quickly. Do **not**
include `## Test plan`, `## How to test`, or similar verification sections — those belong in the PR's CI checks,
not in the body. The PR's CI matrix is the source of truth for what passed.

## Security & Configuration Tips

Do not commit credentials, private API specs, or local machine paths. Keep dependency and license changes intentional,
and run `cargo deny check` plus `cargo machete` when adding, upgrading, or removing dependencies.
